# Design Rationale

## 도메인 책임 분리

이 프로젝트는 메뉴, 포인트, 주문, 인기 메뉴 집계를 한 서비스 안에 구현했지만, 내부 구조는 도메인별 책임이 섞이지 않도록 분리했다.

| 도메인 | 책임 |
| --- | --- |
| `Auth / Member` | 사용자 식별, JWT 인증, 권한 구분 |
| `Menu` | 메뉴 정보, 이미지, 카테고리, 판매 상태 관리 |
| `Point` | 사용자 포인트 지갑, 충전/사용/환불 이력 관리 |
| `Order` | 주문 생성, 주문번호 발급, 주문 상세 스냅샷, 주문 취소 |
| `Ranking` | 최근 7일 인기 메뉴 조회, RDB 집계와 Redis read model 관리 |

서비스가 더 커져 MSA로 분리된다고 가정하면 `Point`와 `Order`는 가장 먼저 분리될 가능성이 크다. 그래서 JPA 연관관계를 강하게 걸지 않고 `memberId`, `menuId`, `orderId` 같은 식별자 참조 중심으로 구현했다.

## 메뉴 설계

메뉴는 온라인 커머스의 재고 상품이 아니라 카페에서 판매하는 메뉴로 보았다. 따라서 재고 수량은 두지 않고, 주문 가능 여부는 `ACTIVE / INACTIVE` 상태로 관리한다.

메뉴 조회에서는 판매중지 메뉴도 내려줄 수 있게 설계했다. 실제 카페 화면에서는 품절 또는 판매중지 메뉴도 노출하고 선택만 막는 경우가 많기 때문이다. 메뉴 삭제는 데이터 추적 가능성을 위해 soft delete로 처리한다.

카테고리는 별도 테이블이 아니라 enum으로 관리했다. 현재 요구사항에서는 카테고리 자체의 CRUD, 정렬 순서, 노출 정책 같은 관리 기능이 필요하지 않고, `COFFEE`, `LATTE`, `TEA`처럼 코드에서 고정된 분류로 충분하다고 판단했다.

## 포인트 설계

결제 수단은 포인트 하나로 제한했다. 포인트는 `PointWallet`에 현재 잔액을 저장하고, `PointHistory`에 충전/사용/환불 이력을 남긴다.

현재 과제에서는 `POST /api/points/charge`로 포인트를 직접 충전하지만, 실제 서비스라면 이 API를 사용자가 임의 호출하는 구조는 적합하지 않다. 실제 운영에서는 PortOne 같은 PG 결제 검증이 완료된 뒤 서버 내부에서 포인트 충전 로직을 호출하는 방식이 더 안전하다.

포인트 증감 로직은 서비스에서 숫자를 직접 수정하지 않고 `PointWallet.charge()`, `PointWallet.use()`, `PointWallet.refund()`에 둔다. 잔액 음수 방지와 양수 금액 검증을 지갑 엔티티 내부에서 수행해 포인트 규칙이 여러 서비스로 흩어지지 않게 했다.

## 주문 설계

주문 생성은 다음 흐름으로 처리한다.

```text
1. 로그인 사용자 확인
2. 주문 요청 항목 검증
3. 중복 menuId 수량 병합
4. 메뉴 조회 및 주문 가능 상태 검증
5. 서버 기준 주문 총액 계산
6. 포인트 차감
7. 주문 저장
8. 주문 상세 저장
9. 주문 완료 이벤트 발행
```

주문 총액은 클라이언트가 보내지 않고 서버가 메뉴 가격과 수량으로 계산한다. 주문 시점의 메뉴명과 가격은 `OrderItem`에 스냅샷으로 저장한다. 메뉴명이 변경되거나 가격이 바뀌더라도 과거 주문 내역은 주문 당시 기준으로 보여야 하기 때문이다.

주문 식별자는 DB PK와 별도로 `ORD-yyyyMMddHHmmss-random6` 형태의 주문번호를 둔다. 내부 저장과 조회에는 `id`를 사용하고, 사용자 화면이나 외부 응답에는 주문번호를 사용할 수 있게 분리했다.

주문 취소는 삭제가 아니라 상태를 `CANCELED`로 변경하고, 결제에 사용한 포인트를 환불한다. 주문 내역은 감사와 집계 기준이 될 수 있으므로 물리 삭제하지 않는다.

## 인기 메뉴 설계

인기 메뉴 요구사항은 "최근 7일간 인기 메뉴 3개"와 "메뉴별 주문 횟수가 정확해야 함"이다. 이를 위해 두 가지 조회 경로를 함께 구현했다.

| 버전 | 방식 | 목적 |
| --- | --- | --- |
| V1 | RDB에서 최근 7일 `PAID` 주문을 집계 | 정합성 기준선 |
| V2 | Kafka 주문 이벤트로 Redis ZSET 점수 갱신 후 조회 | 이벤트 기반 read model 실험 |

V1은 MySQL의 주문/주문상세 원본 데이터를 기준으로 집계하므로 정확성 기준선으로 사용한다. V2는 주문 생성/취소 이벤트를 Kafka로 전달하고 Redis ZSET 점수를 증감해 조회 성능을 개선하는 구조다.

다만 Redis는 원본 저장소가 아니다. V2는 빠른 조회를 위한 read model이고, 정합성 보정이 필요할 경우 V1 집계 결과를 기준으로 Redis 랭킹을 재생성하는 스케줄러를 둘 수 있다.
