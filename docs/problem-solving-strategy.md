# Problem Solving Strategy

## 핵심 문제 정의

이번 과제에서 가장 위험한 지점은 주문과 포인트다. 같은 사용자가 동시에 여러 주문을 요청하면 다음 문제가 생길 수 있다.

```text
보유 포인트: 10,000P
메뉴 가격: 10,000P

요청 A: 잔액 10,000P 확인
요청 B: 잔액 10,000P 확인
요청 A: 주문 성공, 잔액 0P
요청 B: 주문 성공, 잔액 -10,000P 또는 결제 초과 성공
```

따라서 주문 생성과 취소는 단순 CRUD가 아니라 포인트 잔액, 주문 상태, 주문 상세, 인기 메뉴 이벤트가 함께 움직이는 일관성 문제로 보았다.

## 동시성 전략

동시성 제어는 Redis 분산락과 RDB row lock을 함께 사용한다.

```text
Redis 분산락
- 같은 사용자의 주문 생성/취소 요청을 애플리케이션 진입 지점에서 직렬화
- 다중 서버/다중 인스턴스 환경에서도 같은 memberId 기준으로 하나의 락 사용

RDB 비관락
- point_wallets row를 SELECT FOR UPDATE로 잠금
- Redis 장애나 락 누락 상황에서도 포인트 잔액의 최종 정합성을 보장
```

| 기능 | 적용 전략 |
| --- | --- |
| 주문 생성 | `@RedisLock(key = 'lock:order:member:' + memberId)` + 지갑 비관락 |
| 주문 취소 | `@RedisLock(key = 'lock:order:member:' + memberId)` + 지갑 비관락 |
| 포인트 충전 | 지갑 비관락 |
| 포인트 사용/환불 | 지갑 비관락 |

포인트 충전에는 별도 Redis 분산락을 두지 않았다. 충전은 단일 지갑 row의 증가 연산이고, 현재 구현에서는 DB 비관락으로 충분히 직렬화된다. 반면 주문 생성/취소는 포인트 변경과 주문 상태 변경, 이벤트 발행까지 이어지는 더 큰 작업이므로 사용자 단위 분산락을 추가했다.

## 락 전략 비교

| 선택지 | 장점 | 한계 | 최종 판단 |
| --- | --- | --- | --- |
| 락 없음 | 구현이 단순함 | 동시 주문 시 lost update, 초과 결제 위험 | 제외 |
| 낙관락 | 충돌이 적을 때 성능이 좋음 | 결제/주문은 실패 후 재시도 UX와 부수효과 관리가 복잡함 | 보조 장치로만 적합 |
| RDB 비관락 | 최종 잔액 정합성이 강함 | DB row lock 경합 증가 | 포인트 최종 방어선으로 선택 |
| Redis 분산락 | 다중 인스턴스에서 사용자 단위 진입 제어 가능 | Redis만으로 DB 정합성을 보장할 수 없음 | 주문 생성/취소 진입 제어로 선택 |

최종 전략은 `Redis 분산락 + RDB 비관락`이다. Redis 락은 같은 사용자의 요청을 먼저 줄여 DB 경합을 낮추고, RDB 비관락은 포인트 잔액의 최종 정합성을 보장한다.

## 데이터 일관성 전략

원본 데이터는 MySQL에 저장한다.

```text
MySQL
- 회원
- 메뉴
- 포인트 지갑
- 포인트 이력
- 주문
- 주문 상세

Redis
- 메뉴 조회 캐시
- 인기 메뉴 조회용 ZSET
- 분산락

Kafka
- 주문 완료/취소 이벤트 전달
```

Redis와 Kafka는 보조 계층이다. Redis 랭킹이 일시적으로 틀어지더라도 주문과 포인트의 원본 데이터는 MySQL에 남아 있어야 한다. 그래서 인기 메뉴 V1을 RDB 집계 방식으로 유지하고, V2는 Redis 기반 read model로 분리했다.

주문 생성 트랜잭션 안에서는 포인트 차감, 주문 저장, 주문 상세 저장을 함께 처리한다. 주문 이벤트는 Spring `@TransactionalEventListener(phase = AFTER_COMMIT)`를 통해 DB 커밋 이후 Kafka로 전달한다. 트랜잭션이 실패하면 이벤트도 Kafka로 전달되지 않도록 하기 위한 선택이다.

## 주문 이벤트 처리 전략

주문 완료 이후 인기 메뉴 V2 갱신 흐름은 다음과 같다.

```text
OrderService
  -> 주문/포인트 트랜잭션 성공
  -> OrderPaidEvent Spring Event 발행
  -> AFTER_COMMIT listener
  -> Kafka publish
  -> Kafka consumer
  -> Redis ZSET score 증가
```

주문 취소도 같은 구조로 `OrderCanceledEvent`를 발행하고 Redis ZSET 점수를 감소시킨다.

이 구조는 주문 트랜잭션과 Kafka 발행을 직접 강하게 묶지 않는다. 현재 구현은 `AFTER_COMMIT` 이벤트 발행으로 DB 커밋 전 Kafka 발행을 피했다. 운영 수준의 더 강한 보장이 필요하다면 Outbox Pattern을 추가해 `order_outbox` 테이블에 이벤트를 저장하고 별도 publisher가 Kafka 발행을 재시도하는 구조로 확장할 수 있다.

## 인기 메뉴 조회 전략

| 관점 | 선택 |
| --- | --- |
| 정확한 기준선 | V1 RDB 집계 |
| 빠른 조회 실험 | V2 Redis ZSET |

V1은 최근 7일 `PAID` 주문만 대상으로 `order_items.quantity`를 합산한다. 취소된 주문은 제외된다. 이 방식은 DB 부하가 증가할 수 있지만, 언제든 원본 데이터 기준으로 정확한 값을 다시 만들 수 있다.

V2는 주문/취소 이벤트를 Redis ZSET에 반영한다. 일별 key를 사용하고 최근 7일 key를 합산해 Top 3를 계산한다. 현재 구조는 `rolling:7d` 통합 ZSET을 별도로 유지하지 않기 때문에 메뉴 수가 매우 많아지면 조회 비용이 증가할 수 있다.

## 캐시 전략

메뉴 목록은 변경보다 조회가 많은 데이터로 보고 Redis Cache를 적용했다. UI도 전체 메뉴를 항상 가져오는 방식보다 카테고리별 메뉴 조회를 기본 흐름으로 가정했다.

```text
GET /api/menus?category=COFFEE
-> cache key: category:COFFEE
```

메뉴 생성, 수정, 상태 변경, 삭제 시에는 `menus` 캐시를 전체 evict한다. 현재 메뉴 수와 카테고리 수가 크지 않기 때문에 단순한 전체 무효화를 선택했다. 메뉴 수가 많아지면 카테고리 단위 evict로 좁힐 수 있다.

## 다중 인스턴스 전략

다중 서버 환경을 전제로 로컬 메모리에만 의존하는 구조를 피했다.

| 문제 | 전략 |
| --- | --- |
| 서버별 캐시 불일치 | Redis Cache 사용 |
| 서버별 락 불일치 | Redisson 기반 Redis 분산락 사용 |
| 주문 이벤트 전달 | Kafka topic 사용 |
| 최종 원본 정합성 | MySQL 트랜잭션과 row lock 사용 |

애플리케이션 인스턴스가 여러 대로 늘어나도 핵심 상태는 MySQL, Redis, Kafka 같은 외부 인프라를 통해 공유된다.
