# Trade-offs / Limitations

## Redis와 Kafka는 과제 규모 대비 무거울 수 있음

현재 과제 규모만 보면 Redis, Kafka, Redisson을 모두 도입하지 않아도 기능 구현은 가능하다. 메뉴 수와 주문량이 작다면 RDB 집계와 DB row lock만으로도 충분하다.

그럼에도 Redis와 Kafka를 도입한 이유는 다음이다.

- 다수 서버 / 다수 인스턴스 환경에서 공유 락과 공유 캐시를 설계해보기 위해
- 인기 메뉴 집계를 RDB 기준선과 이벤트 기반 read model로 비교하기 위해
- 주문 내역 실시간 전송 요구사항을 Kafka 이벤트로 표현하기 위해
- 기술 도입 자체보다 RDB 원본 정합성을 유지한 채 보조 인프라를 붙이는 방식을 검증하기 위해

따라서 README와 발표에서는 "이 규모에 반드시 필요한 기술"이 아니라 "확장성과 학습 목적을 고려한 설계 실험"으로 설명하는 편이 맞다.

## Redis ranking V2의 한계

현재 V2 인기 메뉴 조회는 일별 ZSET을 유지한다.

```text
popular:menus:yyyy-MM-dd
```

조회 시점에는 최근 7일치 ZSET을 읽고, 메뉴별 점수를 합산한 뒤 Top 3를 계산한다.

장점:

- 주문 이벤트가 발생할 때마다 점수를 누적할 수 있다.
- RDB의 전체 주문 상세를 매번 group by 하지 않아도 된다.
- 취소 이벤트도 점수 감소로 반영할 수 있다.

한계:

- 조회 시점에 7일치 데이터를 합산한다.
- 메뉴 종류가 매우 많아지면 V2 조회 비용도 증가한다.
- Redis 이벤트 누락이나 중복 처리 가능성을 완전히 제거하지는 못한다.
- 최종 정합성 기준은 여전히 RDB V1 집계다.

확장안:

```text
popular:menus:rolling:7d
```

주문/취소 이벤트가 발생할 때 rolling ZSET도 함께 갱신하고, 스케줄러가 RDB 기준 집계 결과로 rolling ZSET을 주기적으로 재생성하면 조회 시점에는 Top 3만 바로 조회할 수 있다.

## Outbox Pattern 미적용

현재 주문 이벤트는 Spring Event `AFTER_COMMIT` 이후 Kafka로 발행한다. 이 방식은 DB 커밋 전에 Kafka 이벤트가 나가는 문제는 피한다.

남는 문제:

- DB 커밋은 성공했지만 Kafka 발행이 실패할 수 있음
- Kafka 발행 실패 이벤트를 재시도할 영속 저장소가 없음

운영 수준 보장이 필요하다면 다음 구조가 더 적합하다.

```text
1. 주문 생성
2. 포인트 차감
3. 주문 저장
4. order_outbox 이벤트 저장
5. 트랜잭션 커밋
6. 별도 publisher/scheduler가 outbox 이벤트를 Kafka로 발행
7. 성공 시 published 처리
```

이번 프로젝트에서는 과제 범위와 구현 복잡도를 고려해 Outbox까지는 적용하지 않고, `AFTER_COMMIT` 이벤트로 트랜잭션 이후 발행 구조를 구현했다.

## AI 추천은 부가 기능임

Spring AI 기반 추천 API는 과제 필수 요구사항이 아니라 학습용 확장 기능이다. 현재는 메뉴 후보가 적기 때문에 RAG나 fine-tuning 대신 실제 판매중 메뉴 후보를 프롬프트에 직접 넣는 방식을 선택했다.

한계:

- 외부 OpenAI 응답 시간에 영향을 받는다.
- 프롬프트만으로는 추천 품질을 완전히 보장할 수 없어 서버에서 후보 필터링과 `menuId` 검증을 수행한다.
- 메뉴 설명, 원두, 알레르기, 영양 정보 문서가 많아지면 RAG 도입을 고려할 수 있다.

## 포인트 충전 API의 현실적 한계

과제 요구사항 때문에 `POST /api/points/charge`를 제공한다. 하지만 실제 결제 서비스에서 사용자가 이 API를 직접 호출해 포인트를 늘릴 수 있으면 안 된다.

실제 운영 구조:

```text
Client
  -> PG 결제 요청
  -> PG 결제 승인
  -> Server가 PG 결제 검증
  -> 검증 성공 시 내부적으로 포인트 충전
```

현재 구조는 과제 요구사항과 동시성 검증을 위한 단순화다.

## 메뉴 카테고리 enum의 한계

카테고리는 enum으로 관리한다.

장점:

- 현재 요구사항에서는 단순하고 빠르다.
- 잘못된 카테고리 입력을 코드 레벨에서 제한하기 쉽다.
- 카테고리별 조회 캐시 key를 단순하게 유지할 수 있다.

한계:

- 운영자가 카테고리를 동적으로 추가/수정할 수 없다.
- 카테고리 노출 순서나 아이콘 같은 메타데이터를 관리하기 어렵다.

카테고리 관리 기능이 요구되면 별도 `menu_categories` 테이블로 분리하는 편이 낫다.

## JPA 연관관계 최소화의 한계

현재 엔티티는 `Order -> OrderItem -> Menu`, `Member -> PointWallet` 같은 객체 연관관계를 강하게 맺지 않고 ID 참조를 사용한다.

장점:

- 도메인 경계를 느슨하게 유지할 수 있다.
- MSA 분리 가능성을 설명하기 쉽다.
- 불필요한 지연 로딩과 N+1 가능성을 줄일 수 있다.

한계:

- 객체 그래프 탐색 편의성이 낮다.
- 필요한 데이터는 repository query나 reader를 통해 명시적으로 가져와야 한다.
- 작은 단일 모놀리식 프로젝트에서는 코드가 다소 장황해 보일 수 있다.

이번 프로젝트는 과제의 "다수 서버 / 확장성 / 일관성" 설명을 우선해 ID 참조 방식을 선택했다.

## 성능 테스트의 한계

성능 테스트는 로컬 Docker 환경 기준이다.

주의:

- 운영 DB/Redis/Kafka 네트워크 환경과 다를 수 있다.
- 테스트 데이터는 실제 사용자 트래픽 분포를 완전히 반영하지 않는다.
- 측정 결과는 절대 성능 수치가 아니라 방식 간 비용 구조 비교로 보는 것이 맞다.
